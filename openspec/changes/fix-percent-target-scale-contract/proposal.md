## Why

`bfh_qic()` passes `target_value` unchanged to `qicharts2::qic()` regardless of `y_axis_unit` or `multiply` parameters (`R/bfh_qic.R:673-675`). When users with `y_axis_unit = "percent"` (default `multiply = 1`) intuitively pass `target_value = 2.0` (meaning 2%), qicharts2 plots the target line at y = 2.0 — which renders as 200% on the percent-formatted axis. Clinically misleading: a target appears unmet when it was actually met.

This is the most common intuitive misunderstanding of the percent contract, and the broken pattern is replicated in `README.md:81-91` (currently autogenerated, not yet authoritative). Codex code review (2026-04-27, finding #1) flagged this as CRITICAL.

## What Changes

- **BREAKING**: `bfh_qic()` validates `target_value` against an explicit scale derived from `y_axis_unit` and `multiply`. Out-of-range values raise an error with actionable migration hint.
- Define explicit contract:
  - `y_axis_unit = "percent"` + `multiply = 1` → `target_value` SHALL be in `[0, 1.5]` (proportion)
  - `y_axis_unit = "percent"` + `multiply = 100` → `target_value` SHALL be in `[0, 150]` (percent)
  - `y_axis_unit = "percent"` + arbitrary `multiply = m` → `target_value` SHALL be in `[0, m * 1.5]`
  - Other `y_axis_unit` values → no scale validation (subject to existing numeric validation)
  - Negative `target_value` SHALL raise error for `y_axis_unit = "percent"`
  - Contract SHALL apply uniformly across all chart types including run-chart
- Update `README.md` example to use `target_value = 0.02` (matching `target_text = "↓ Target: 2%"`)
- Add `@details` "Percent Target Contract" section to `bfh_qic()` Roxygen
- Add 8 new tests covering both contract paths, error path, negative target, boundary values, run-chart parity, and non-percent skip

```r
  as.numeric(input$target_value)

bfh_qic(..., y_axis_unit = "percent", target_value = target_proportion)
```
